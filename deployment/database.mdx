Car Bot uses PostgreSQL as its production database, accessed through [Drizzle ORM](https://orm.drizzle.team). Schema definitions live in `shared/schema.ts` and are shared between the server and client. Validation schemas are generated automatically from the Drizzle table definitions using `drizzle-zod`, so the database schema, TypeScript types, and Zod validators are always in sync. Schema migrations are applied with `drizzle-kit` via a single npm script.

## Tech stack

- **PostgreSQL** — Relational database accessed via the `pg` driver
- **Drizzle ORM** — Type-safe query builder and schema definition
- **drizzle-zod** — Derives Zod validation schemas directly from Drizzle table definitions
- **drizzle-kit** — CLI tool for schema migrations (`npm run db:push`)

## Users table

The `users` table is defined in `shared/schema.ts`. The primary key is a UUID generated by PostgreSQL's `gen_random_uuid()` function, so no application-level ID generation is required on insert.

```typescript shared/schema.ts
import { sql } from "drizzle-orm";
import { pgTable, text, varchar } from "drizzle-orm/pg-core";
import { createInsertSchema } from "drizzle-zod";
import { z } from "zod";

export const users = pgTable("users", {
  id: varchar("id").primaryKey().default(sql`gen_random_uuid()`),
  username: text("username").notNull().unique(),
  password: text("password").notNull(),
});

export const insertUserSchema = createInsertSchema(users).pick({
  username: true,
  password: true,
});

export type InsertUser = z.infer<typeof insertUserSchema>;
export type User = typeof users.$inferSelect;
```

`insertUserSchema` omits `id` from the insert shape — the database generates it. `User` represents a full row as returned by a `SELECT`, including `id`.

## Storage interface

All database access goes through the `IStorage` interface defined in `server/storage.ts`. This abstraction makes it straightforward to swap the underlying store (for example, replacing `MemStorage` with a Drizzle-backed PostgreSQL implementation) without changing any route logic.

```typescript server/storage.ts
export interface IStorage {
  getUser(id: string): Promise<User | undefined>;
  getUserByUsername(username: string): Promise<User | undefined>;
  createUser(user: InsertUser): Promise<User>;
}
```

See [Server API reference](/deployment/api-reference) for parameter and return type details on each method.

## Default in-memory implementation

`MemStorage` implements `IStorage` using a `Map<string, User>` held in process memory. It is the default export from `server/storage.ts` and requires no external dependencies.

```typescript
export class MemStorage implements IStorage {
  // Uses Map<string, User> internally
  // createUser generates UUIDs via Node's crypto.randomUUID()
}

export const storage = new MemStorage();
```

<Note>
  `MemStorage` loses all data when the process restarts. It is suitable for local development and testing only. Production deployments must set `DATABASE_URL` and use a PostgreSQL-backed storage implementation.
</Note>

## Applying schema changes

Use `drizzle-kit push` to apply your Drizzle schema directly to the database without generating migration files. This is the recommended workflow during active development.

```bash
npm run db:push
```

This command reads `DATABASE_URL` from your environment and pushes the current schema defined in `shared/schema.ts` to the connected PostgreSQL database.

## Connection string format

`DATABASE_URL` must be a valid PostgreSQL connection string in the following format:

```
postgresql://user:password@host:5432/dbname
```

| Segment | Description |
|---|---|
| `user` | PostgreSQL role name |
| `password` | Role password |
| `host` | Database server hostname or IP |
| `5432` | Default PostgreSQL port |
| `dbname` | Target database name |

For hosted databases (Neon, Supabase, Railway), copy the connection string directly from the provider's dashboard. If your provider offers a pooled connection string, prefer that to avoid exhausting connections on Vercel's serverless functions.
